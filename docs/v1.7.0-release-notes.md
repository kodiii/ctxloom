# ctxloom v1.7.0 — graph quality, language reach, and the production wedge

**Release date**: 2026-05-22
**Theme**: methodology rigor + breadth without losing depth

This release closes the v1.7.0 milestone we set at the start of the cycle:
ship a benchmark-validated graph engine, support every meaningful AI
coding host without auto-detection rot, and make the indexer survive
50k+ file monorepos. Plus a handful of correctness fixes for hosts
whose vendor docs had drifted out from under us.

## TL;DR — what we can defend with numbers

Measured on a 5-repo × 3-PR external-oracle bench (PRs from
expressjs/express, tiangolo/fastapi, pallets/flask, gin-gonic/gin,
encode/httpx — 15 PRs total, ground truth = the merged diff):

| Metric                | v1.5.x | **v1.7.0**       | Notes |
|-----------------------|-------:|------------------:|-------|
| **Graph reachability** | 0.81  | **0.94**         | +0.13 from Go resolver fan-out (#189) |
| **Source recall**      | 0.56  | **0.61**         | +0.05 |
| **Avg F1**             | 0.40  | **0.42**         | precision-floor moves slowly; this is honest |
| **Symbol coverage**    | 1.00  | **1.00**         | AST-declared symbols indexed with correct file attribution |
| **Import coverage**    | 1.00  | **1.00**         | AST-found relative imports resolved to graph edges |
| **Token reduction**    | (estimated 92%) | **24.6× (~96%) measured** | same Skeletonizer the production tools serve |

**Embedding upgrade path** (opt-in via `CTXLOOM_EMBEDDING_MODEL=jina-code`):
72.5% better discrimination on code-similarity queries than the legacy
MiniLM default. Same local runtime, same privacy story, +50 MB on disk.

**Languages with import resolution + symbol indexing**: 18 (was 13).
**MCP host adapters**: 17 (was 13) with corrected paths for OpenCode,
Continue, and Codex — three hosts the old code wrote configs to that
the vendors silently ignored.

## What shipped (by theme)

### Bench rigor (PRs #186-191, #194)

The whole release is gated on a benchmark that measures graph quality
against an **external oracle** (the merged PR diff from GitHub), not
the graph's own traversal. The competitor approach — using the graph
itself as ground truth — is tautological; you can't measure correctness
against your own predictions.

Four new audits run per PR:

- **Symbol declaration coverage** (#186) — fraction of AST-declared
  function/class/method/interface symbols that appear in the graph's
  symbolIndex with correct file attribution. The primary test of the
  "absurd accuracy across all project files" claim.
- **Import edge coverage** (#187) — fraction of AST-found intra-repo
  imports that resulted in a graph forwardEdge. Direct measure of the
  import resolver's correctness, per-extension breakdown isolates
  language-specific gaps.
- **Graph reachability** (already in v1.6.x) — BFS over the import
  graph from the entry point, measures how much of the source-file
  ground truth is structurally reachable.
- **Token reduction** (#191) — naive baseline (full files for the
  diff + 1-hop imports) vs ctxloom's skeleton view of the predicted
  files. Both sides use the SAME production token estimator and the
  SAME production Skeletonizer end users hit. No synthetic-bench
  cheating.

### Graph correctness fixes (PRs #189, #199)

- **Go package fan-out** (#189) — pre-fix, a `import "github.com/foo/bar/pkg"`
  statement only created an edge to ONE .go file in `pkg/`. Go packages
  are compiled together; the whole package is the import target. Now
  one statement creates edges to every non-test .go file in the package
  directory plus bidirectional test↔source edges. Lifts gin's
  graphReachability from **0.32 → 0.95** (~3× improvement).
- **FD baseline** (#199) — the FileWatcher and the indexer kept SEPARATE
  ignore lists that silently drifted. A `.vscode-test/Visual Studio Code.app/...`
  bundle in your repo (very common in VS Code extension projects)
  would push the MCP server past macOS's 256-FD default at boot.
  Single source of truth + function-based ignore predicate now prevents
  drift permanently.

### Languages (PR #193)

Five new languages with import resolution + symbol indexing: C/C++,
Scala, Lua, Elixir, Zig. Brings the total to 18 with deep support
(not just "tree-sitter grammar loaded").

Each language's resolver was researched against vendor convention,
not transplanted from third-party PLATFORMS dicts that had bugs in
two thirds of our verified hosts. Path-traversal guards on every
relative-style resolver; greedy-regex bug found and fixed in the
Scala extractor (would otherwise emit malformed `com.example.`
specifiers on brace-grouped imports).

### MCP hosts (PRs #195, #196, #197)

Four new hosts (Zed, Gemini CLI, Kiro, OpenCode) plus corrected paths
for three silently-broken existing ones:

- **OpenCode**: config key is `mcp`, NOT the conventional `mcpServers`.
- **Continue**: workspace-scoped per-server YAML at
  `.continue/mcpServers/<name>.yaml`, NOT the historical
  `~/.continue/config.json`. Required extending the host-adapter
  contract with a `customWriter` hook.
- **Codex**: TOML at `config.toml` with snake_case key `mcp_servers`,
  NOT JSON at `mcp.json` with `mcpServers`. Required a shared-file
  capable `customWriter` + a `customRemove` hook so uninstall surgically
  removes our block without clobbering the rest of the user's config.

Auto-detection in `ctxloom setup` covers all 17 hosts; project-scoped
paths lead the lookup order for hosts where the per-project install
is the canonical workflow (Cursor, Gemini CLI, OpenCode).

### Monorepo support (PR #198)

The indexer's old `collectFiles → for (file in files) upsert` flow
stalled on Next.js at ~18%. Two rewrites:

- **Streaming file walk** — async generator yields files as discovered
  instead of materializing the full list. First file ships to the
  upsert path within ~50 ms instead of after the 5-15s upfront pass.
- **Batched LanceDB upserts** — 50 file records per transaction pair
  (was 2 transactions per file). On 50k-file repos: 100,000 → ~2,000
  transactions, ~50× fewer manifests written, ~5× faster wall time,
  FD ceiling stops mattering.

### Embedding acceleration (PR #204)

Step 1 of a three-step acceleration plan. ONNX runtime session
overhead is paid PER CALL; batching N file embeddings into one
inference call amortizes that cost. **3-10× speedup** depending on
model size (larger models benefit more — session-setup cost is a
larger fraction of per-call wall time).

This is the no-tradeoff win: same UX, same privacy, same model, just
stop paying the per-file overhead.

### Pluggable embedding model (PR #192) + jina-code micro-bench (#194)

`CTXLOOM_EMBEDDING_MODEL=jina-code` opts into a code-specific
embedding model that's measured to be **+72.5% better at code-semantic
discrimination** than the legacy MiniLM default. Same local
`@huggingface/transformers` runtime, +50 MB on disk, +384 vector
dimensions (LanceDB index gets a marker file so swapping models on
an existing index throws a clear `re-index required` error rather
than silently corrupting the table layout).

Kept opt-in for v1.7.0; will promote to default in v1.8.0 once we
have full F1 numbers comparing both models on the same external
oracle.

## What's intentionally NOT in this release

- **Ollama backend** — drafted, deferred to v1.8.0. Lots of users
  already run Ollama; Metal/CUDA acceleration on top of jina-code
  would unlock another 2-5× on indexing wall time. The pluggable-model
  scaffolding from #192 already paves the path.
- **Cloud embedding APIs (Ollama Turbo, Voyage-code-3)** — drafted
  and rejected. Both would ship every file in your codebase to a
  third-party cloud. Breaks the "fully local, no network at inference
  time" promise that's core to ctxloom's pitch. Will reconsider when
  a paying enterprise customer needs the throughput and waives the
  privacy concern in writing.
- **Worker-thread parallelism** — Step 2 of the acceleration plan;
  shipping after #204 is validated in the next bench run.

## Methodology hygiene we're committed to

- **Token estimator unified** between the production budget surface
  and the bench. Numbers reported here line up with what end users
  see in their `max_response_tokens` telemetry — no synthetic-bench
  tokenizer that no production code touches.
- **Bench corpus pinned** to 5 repos × 3 PRs of merged real-world
  PRs, hand-selected against methodology rules: ≥2 source files, not
  a dep bump, ≥4 month temporal spread, has test changes.
- **External oracle** for every metric except symbol/import coverage
  (which are direct AST-vs-graph comparisons with no oracle needed).
- **All numbers reproducible**: `npm run bench:full` on commit
  `<sha>` produces the same `evaluate/reports/summary.md` we publish.

## Migration notes

**For most users**: nothing to do. `ctxloom-pro@1.7.0` installs over
your existing version and re-indexing is automatic on first use.

**If you opt into jina-code**: set `CTXLOOM_EMBEDDING_MODEL=jina-code`
and run `ctxloom vectors-cleanup --reset` once to invalidate the
MiniLM-shaped LanceDB table. Re-index will rebuild with the 768-dim
schema.

**If you use Continue, Codex, or OpenCode**: re-run `ctxloom setup`
once. The corrected MCP config paths land where the vendors actually
read from now.

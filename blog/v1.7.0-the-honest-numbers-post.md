# The honest-numbers post (ctxloom v1.7.0)

*A code-context engine you can actually measure.*

---

A few months ago I went looking for a way to measure whether ctxloom
actually does what it claims — give an AI coding agent enough
structural context that it doesn't re-read every file to figure out
what's going on.

What I found in the surrounding ecosystem made me uncomfortable.

Most code-context tools publish big numbers — "23,000× token
efficiency," "92% reduction," "5× faster" — and when you go look at
how they're measured, the answer is almost always: **the tool measures
itself against itself**. Tool X predicts which files are affected by
a change. Tool X also traverses its own graph to define which files
were "supposed" to be affected. Then it compares prediction to graph
and announces the result.

That's a tautology. If you train a student to mark their own exam, of
course they get an A.

So I built a bench with an external oracle.

## How the bench works

For each of 15 real, merged pull requests across 5 popular repos (Express,
FastAPI, Flask, Gin, HTTPX), we run this:

1. Take the most-modified file in the PR — the file a human reviewer
   would naturally start reading.
2. Ask ctxloom: *"given a change here, what other files are affected?"*
3. Compare ctxloom's prediction against the **actual** ground truth:
   the files in the merged PR diff.

The ground truth doesn't come from ctxloom. It comes from GitHub. It's
what the human author actually changed.

Then we report F1, precision, recall, and a separate "graph
reachability" number that measures whether the right files even
exist in ctxloom's graph (independent of how it ranks them).

## The honest numbers, v1.7.0

| Metric | Value |
|--------|-------|
| Avg F1 | **0.42** |
| Avg source recall | **0.61** |
| Avg graph reachability | **0.94** |
| Avg symbol coverage | **1.00** |
| Avg import coverage | **1.00** |
| Avg token reduction | **24.6×** |

That F1=0.42 is not a press-release number. It's a real number. It
means: of the files we predict as affected, 47% are actually in the
PR diff, and we find 52% of the files the human author changed. The
remaining gap is the work — we haven't published an "F1 = 0.94"
number because there isn't one.

The numbers that ARE high are the ones that mean what they sound
like:

- **graphReachability = 0.94** — given the entry-point file, 94% of
  the source files in the human's PR are reachable via BFS over our
  import graph. The graph contains them; the prediction algorithm
  just has to be smart enough to surface them.
- **symbolCoverage = 1.00** — every function/class/method/interface
  the AST parser finds in your code appears in our symbol index with
  the correct file attribution. The graph genuinely knows where
  everything is defined.
- **importCoverage = 1.00** — every relative import the AST finds
  resolves to a real graph edge. The import resolver isn't dropping
  edges silently.
- **24.6× token reduction** — measured against the production token
  estimator (the same one your `max_response_tokens` budget surface
  uses) and the production Skeletonizer (the same code path
  `ctx_get_context_packet` serves). No synthetic-bench tokenizer
  invented to make the number look better.

## What was hard

Three things made the bench painful enough that we almost gave up
twice:

### 1. The Go resolver was structurally wrong

A Go import like `import "github.com/foo/bar/pkg"` brings the entire
`pkg/` directory into your file's namespace — every .go file in that
folder is part of the same compilation unit. Our resolver returned
ONE file (the alphabetically-first). For Gin's `binding/` package
(~20 files), 95% of the package was invisible to the graph from any
caller's perspective.

This dragged gin's graphReachability to **0.32**. It looked like our
graph was broken on Go specifically. Turns out the graph was fine;
the resolver was unintentionally pretending Go works like JavaScript.

Fix: fan out one import statement to N edges (one per non-test .go
file in the target package), plus bidirectional edges between
`foo.go` and `foo_test.go` to model Go's same-package test convention.
Gin's graphReachability jumped to **0.95**. ~3× improvement.

This is the kind of thing you only find when you measure against an
external oracle. If we'd used the graph as its own oracle, gin would
have looked fine — because the graph would have been comparing its
broken prediction against its own broken model of reality.

### 2. Three host adapters were writing config to the wrong place

`ctxloom setup` auto-detects which AI coding tools you have installed
(Cursor, Continue, Codex, Zed, Gemini CLI, Kiro, OpenCode, Claude
Code, Claude Desktop, Windsurf, VS Code Copilot, ...) and registers
ctxloom as an MCP server in each one's config.

Three of the registry entries we inherited from a community-maintained
catalog were silently broken on current vendor versions:

- **OpenCode**: the schema key is `mcp`, not `mcpServers`. Wrong key,
  silently ignored.
- **Continue**: the config layout moved from a JSON file with an
  array entry to per-server YAML files at
  `.continue/mcpServers/<name>.yaml`. Writing the old JSON path
  silently failed.
- **Codex**: the file format changed from JSON to TOML, AND the path
  changed from `mcp.json` to `config.toml`, AND the schema key from
  `mcpServers` to `mcp_servers`. Three independent corrections,
  every one of them silent on current Codex.

Users got a `✓ Added ctxloom to <host>` success message and then
ctxloom never appeared in the host's tool list. We fixed all three
in v1.7.0 — each with regression tests that pin the wrinkle so a
future "cleanup" refactor can't accidentally undo it.

### 3. Indexing 50k+ file monorepos stalled at ~18%

We tried to bench Next.js and the indexer never finished. We tried
multiple times. We dropped Next.js from the corpus.

Root cause: every file did its own LanceDB upsert (one delete + one
add = 2 transactions). For a 50k-file repo that's 100,000
transactions, each writing a manifest + transaction file. On macOS
the per-process FD limit is 256 by default. We were hitting it
~10,000 FDs/sec and the indexer was stalling on file-descriptor
contention.

Two changes in v1.7.0:

- **Streaming file walk** — async generator yields files as they're
  discovered. Indexer starts processing in parallel with the walk.
- **Batched upserts** — 50 records per LanceDB transaction pair.
  100,000 → ~2,000 transactions. 50× fewer manifests, ~5× faster
  wall time, FD ceiling stops mattering.

Then, since we were already in the indexer's hot path: **batched ONNX
inference**. ONNX runtime pays a per-call session overhead; batching
50 file embeddings into one inference call amortizes that overhead.
Another 3-10× on the embedding step.

## What we won't do

While building v1.7.0 we surveyed every popular code-context tool I
could find — code-review-graph, Graphify, Augment, Cody, and a few
others. The pattern that's most tempting (and most dangerous) is
**ship embedding to the cloud for speed**.

It works. You can get 50-100× speedup over local inference by sending
every file to Voyage, OpenAI, or a hosted Ollama. The catch is that
every line of your private code now goes over the wire to a third
party.

We're not going to do that.

ctxloom's core pitch is **fully local, no network at inference time**.
Enterprise users — the people who pay for ctxloom-pro — explicitly
don't want their code in someone else's cloud. We're committed to that
constraint even when there's a free 50× win lying on the table by
breaking it.

(If a paying customer needs the throughput and waives the privacy
concern in writing, we'll build the cloud backend. Until then, we
won't.)

## Where we're not great yet

A few things I'd flag if you're evaluating ctxloom against a competitor
that publishes prettier numbers:

- **F1 = 0.42 means we miss things.** A real PR's affected-file set is
  often broader than any structural graph can predict — the author
  changes a CHANGELOG.md, a YAML config, a docs file. Those aren't
  reachable from the entry point because they're not part of the
  dependency graph. We report a separate **source recall** (filtered
  to indexable files) that's higher — 0.61 — and we're working on
  the gap between 0.42 and 1.0 honestly.
- **TypeScript/JS/Python/Go are deep; the rest are good-enough.** Out
  of 18 supported languages, only 4 have benchmark coverage. The
  others have import resolution and symbol indexing, but the F1
  numbers above are computed only on the 5-repo corpus (which spans
  TS, Python, and Go).
- **The default embedding model is general-purpose.** MiniLM-L6-v2
  is a 2020 English sentence embedder; it works fine on prose but
  treats code like text. We ship `jina-embeddings-v2-base-code` as
  an opt-in (`CTXLOOM_EMBEDDING_MODEL=jina-code`). It's measurably
  72.5% better at distinguishing semantically similar code on a
  10+10 pair micro-bench. We'll promote it to default in v1.8.0
  once we have full F1 numbers comparing both on the same corpus.

## How to try it

```bash
npm install -g ctxloom-pro
ctxloom setup       # auto-detects every AI host you have installed
ctxloom init        # registers ctxloom in this project's .mcp.json
```

`ctx_get_minimal_context(task="explore this codebase")` from your
agent gets you ~150 tokens of orientation and a suggested next tool.

Reproduce every number above:

```bash
git clone https://github.com/kodiii/ctxloom
cd ctxloom
npm install
npm run bench:full
```

Output lands in `evaluate/reports/summary.md`. Same numbers, same
methodology, same external oracle.

If you want to read the methodology in detail, every assumption is
documented in `evaluate/methodology.md` and `evaluate/limitations.md`.
If you want to argue with a number, the bench is in `scripts/bench/`
and the corpus is in `scripts/bench/corpus.ts` — open a PR.

The goal isn't to win a press-release showdown. The goal is to be the
code-context engine you can actually measure.
